01

Project Overview

NoCarry is a full-stack SaaS web application that solves the fairness problem in university group projects. It tracks individual contributions, enables anonymous peer reviews, surfaces freeloaders with hard data, and gives professors AI-generated grading reports — all in one platform.

Who uses it

RoleWhat they do
StudentCreate/join projects, manage tasks via Kanban, review teammates, track personal contribution score
Team LeaderAll student features + create projects, invite members, set deadlines, upload files
ProfessorCreate courses, link projects, monitor all teams, generate AI grading reports, detect freeloaders

Core loop

  1. Professor creates a course → students create projects inside it
  2. Team leader invites members → members get tasks assigned on a Kanban board
  3. Every action (task created, completed, file uploaded) logs an ActivityLog and increments contribution points
  4. At deadline, projects auto-archive → professor pulls AI grading report with contribution breakdown + peer reviews

Theme System

The theme system is 100% CSS — no JS runtime cost. It's built on CSS custom properties scoped to a data-theme attribute on <html>.

How it works

  1. Root layout injects an inline script that reads localStorage.getItem("nc-theme") and sets document.documentElement.dataset.theme before first paint — zero flash
  2. Every component reads from var(--th-*) variables
  3. ThemeProvider React context exposes theme + setTheme()
  4. FloatingThemeToggle renders in the corner on all pages

Available themes

light dark gruvbox nord tokyo-night dracula catppuccin

CSS variable reference

VariablePurpose
--th-bgPage background
--th-cardCard / panel background
--th-textPrimary text
--th-text-2Secondary / muted text
--th-borderBorder color
--th-accentPrimary action color (buttons, links, active states)
--th-accent-fgText on top of accent (e.g. button label)

Adding a new theme

In globals.css, add a new block:

[data-theme="mytheme"] {
  --th-bg:        oklch(0.15 0.02 250);
  --th-card:      oklch(0.20 0.02 250);
  --th-text:      oklch(0.95 0 0);
  --th-text-2:    oklch(0.65 0 0);
  --th-border:    oklch(0.30 0.02 250);
  --th-accent:    oklch(0.70 0.20 180);
  --th-accent-fg: oklch(0.10 0 0);
}

Then add "mytheme" to the themes array in ThemeProvider.tsx.

Database Schema (Prisma)

Database: PostgreSQL with PgBouncer connection pooling. ORM: Prisma 7.4.1 with @prisma/adapter-pg.

The Prisma singleton in src/lib/prisma.ts uses a pg.Pool (max 3 connections) and the PgBouncer adapter. This prevents connection exhaustion in serverless/edge environments where each request would otherwise open a new DB connection.

Models

User

  • id — UUID (mirrors Supabase auth user ID)
  • email — unique
  • name, preferredName, avatarUrl
  • role — STUDENT | PROFESSOR | TEAM_LEADER
  • school, major, bio
  • githubUrl, linkedinUrl, personalLinks (JSON array)
  • status, statusExpiresAt — custom status message with auto-expiry

Project

  • id, name, description
  • ownerId — the team leader who created it
  • courseId / courseCode — optional link to a Course
  • deadline — DateTime; triggers auto-archive
  • status — ACTIVE | ARCHIVED | DELETED
  • archivedAt, archivedReason

Task

  • projectId, creatorId, assigneeId
  • title, description
  • status — TODO | IN_PROGRESS | DONE
  • dueDate, completedAt
  • outputType — PDF | PPTX | TXT | DOCX | CODE
  • outputFileUrl, outputFileName — deliverable upload

ActivityLog

  • userId, projectId, taskId (optional)
  • action — string like "TASK_CREATED", "TASK_COMPLETED", "FILE_UPLOADED"
  • metadata — JSON for extra context (e.g., task name)
  • This table powers contribution scoring — every meaningful action is logged here

PeerReview

  • projectId, giverId, receiverId — unique compound
  • quality, communication, timeliness, initiative — all Int (1–5)
  • comment — optional anonymous text

Subscription

  • userId — one subscription per user
  • stripeCustomerId, stripeSubscriptionId, stripePriceId
  • plan — FREE | PRO
  • status — active | trialing | past_due | canceled | incomplete
  • currentPeriodEnd — used to show renewal date

ChatMessage + ChatReaction

  • Project-scoped chat — each message has projectId, senderId, text, optional imageUrl
  • ChatReaction — messageId + userId unique, emoji (default 👍)
  • ChatReactionPreference — per-user per-project preferred quick-reaction emoji

ProjectInvite

  • token — unique UUID, sent in the invite email URL
  • email, role, expiresAt, usedAt
  • When a user signs up with a matching email, invites are auto-consumed and they're added to the project

Common Prisma patterns used in this codebase

// Upsert subscription on Stripe webhook
await prisma.subscription.upsert({
  where: { userId },
  create: { userId, plan: "PRO", status: "active", ... },
  update: { plan: "PRO", status: "active", ... },
});

// Soft delete — set status instead of deleting
await prisma.project.update({
  where: { id: projectId },
  data: { status: "DELETED" },
});

// Compound unique check
await prisma.peerReview.findUnique({
  where: { projectId_giverId_receiverId: { projectId, giverId, receiverId } },
});

Authentication Flow

Auth is handled entirely by Supabase Auth with SSR cookie-based sessions. There is no custom JWT logic.

How it works end-to-end

  1. User submits email + password on /login
  2. Browser Supabase client calls supabase.auth.signInWithPassword()
  3. Supabase returns a session — the @supabase/ssr package stores this in an httpOnly cookie automatically
  4. On every server request, createClient() from src/lib/supabase/server.ts reads the cookie and returns the authenticated session
  5. Dashboard layout calls supabase.auth.getUser() — if no session, redirect to /login

Two Supabase clients

FileUsed inPurpose
lib/supabase/server.tsServer Components, API routes, layoutReads cookies, validates session server-side
lib/supabase/client.tsClient Components (login/signup forms)Browser-side sign in, sign up, sign out

User creation flow

// POST /api/auth/register
1. supabase.auth.admin.createUser({ email, password })  // Creates Supabase auth user
2. prisma.user.create({ id: supabase_uid, email, name, role })  // Mirror in DB
3. Check for pending invites matching email → auto-consume them
4. Return user

Logout

// POST /api/auth/logout (form action in MobileNav / Sidebar)
await supabase.auth.signOut()
redirect("/login")
Important: The Prisma User.id is the same UUID as the Supabase auth user ID. This allows cross-referencing without extra joins — always use the Supabase user ID as the Prisma user ID.

Dashboard Layout

src/app/dashboard/layout.tsx is a server component that acts as the auth guard and shell for all /dashboard/* routes.

What it does on every request

  1. Calls supabase.auth.getUser() — redirects to /login if unauthenticated
  2. Fetches prisma.user by Supabase UID — redirects if user not found in DB
  3. Fetches prisma.subscription — determines isPro flag

Billing — Stripe

Plans

PlanProjectsAI ReportsPrice
FREELimited (enforced via plan.ts)No$0
PROUnlimitedYesStripe Price

Checkout flow

  1. User clicks upgrade on /dashboard/plan
  2. Frontend POSTs to /api/stripe/checkout
  3. Server creates a Stripe Checkout Session with success_url + cancel_url
  4. User is redirected to Stripe's hosted checkout page
  5. On success, Stripe fires checkout.session.completed webhook
  6. Webhook handler at /api/stripe/webhook upserts the Subscription record in Prisma

Webhook events handled

EventAction
checkout.session.completedUpsert subscription, set plan=PRO, status=active
customer.subscription.updatedSync status changes (trialing, past_due, active)
customer.subscription.deletedSet plan=FREE, status=canceled
invoice.payment_failedSet status=past_due, notify user
Webhook security: The webhook route reads the raw request body and verifies the stripe-signature header using STRIPE_WEBHOOK_SECRET. Never parse the body as JSON before this check — Stripe requires the raw bytes.

AI — Anthropic Reports

Professors can generate an AI-written grading report for any project. This hits POST /api/projects/[id]/report.

What gets sent to the AI

  • Project name, description, deadline
  • All members with their contribution scores and breakdowns
  • All peer review scores per member (quality, communication, timeliness, initiative) + comments
  • Task list with statuses and assignees

What comes back

A structured text report with:

  • Per-member contribution summary
  • Suggested grade ranges based on data
  • Flags for low-contribution members
  • Peer review sentiment analysis
  • Professor-facing recommendations
PRO only: The API route checks isPro before calling Anthropic. If the user is on FREE, it returns a 403 with an upgrade prompt.

Performance & Security

Performance

  • Server Components by default — only client components that need state/effects are marked "use client"
  • Image optimization — Next.js serves AVIF/WebP via <Image />, 1-year CDN cache
  • Bundle splitting — Three.js components are dynamically imported, not in the initial JS bundle
  • CSS animations on touch devices disabled — @media (pointer: coarse) removes heavy animations on mobile
  • ISR caching — public pages (pricing, terms, changelog) revalidate every 60s with 1hr stale
  • No animation on preference — respects prefers-reduced-motion
  • Prisma connection pooling — max 3 connections via PgBouncer prevents serverless connection storms

Security

  • Auth guard in layout — dashboard/layout.tsx redirects unauthenticated users server-side (not client-side)
  • Supabase RLS — Row-Level Security on all tables; service role key only used server-side
  • Stripe webhook signature — every webhook validated with stripe.webhooks.constructEvent(rawBody, sig, secret)
  • HTTP security headers on all routes: X-Frame-Options, nosniff, strict referrer, Permissions-Policy
  • No secrets in client code — only NEXT_PUBLIC_ prefixed vars reach the browser
  • Soft deletes — projects and courses are never hard-deleted, only status-flagged

Extending the App

Adding a new dashboard page

  1. Create src/app/dashboard/yourpage/page.tsx
  2. It automatically inherits the auth guard and shell from dashboard/layout.tsx
  3. Add a nav link in src/app/dashboard/_components/NavLinks.tsx (and MobileNav.tsx for mobile)

Adding a new API route

  1. Create src/app/api/yourroute/route.ts
  2. Export named functions: export async function GET(req: Request) {}
  3. Always validate session first: const { data: { user } } = await supabase.auth.getUser()
  4. Return NextResponse.json(data) or NextResponse.json({ error }, { status: 4xx })

Adding a new Prisma model

  1. Edit prisma/schema.prisma
  2. Run npx prisma migrate dev --name add_yourmodel
  3. Run npx prisma generate — Prisma client types update automatically

Adding a new theme

  1. Add [data-theme="name"] { --th-* variables } block to globals.css
  2. Add "name" to the themes array in ThemeProvider.tsx
  3. Add a swatch in SettingsModal.tsx

Adding a Pro feature

  1. In the API route, fetch the subscription: const sub = await prisma.subscription.findUnique({ where: { userId } })
  2. Check: const isPro = sub?.plan === "PRO" && (sub.status === "active" || sub.status === "trialing")
  3. Return 403 if not pro, with a descriptive error message
  4. In the UI, use isPro prop (already passed down from layout) to show/hide upgrade prompts
Dev tip: When in doubt about where something renders, search for the CSS class or component name in globals.css — all custom styles are namespaced with nc- (NoCarry prefix), making them easy to find and trace.
